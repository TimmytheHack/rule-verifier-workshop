# 前端 Workbench 重排设计

## 背景

当前前端已经能展示 `WorkbenchResponse`、上传数据集并调用后端 API，但主界面存在几个直接影响使用的问题：

- 首页默认展示 `demo_run.json`，用户容易把 mock 结果误认为真实查询结果。
- 主查询页采用三栏满屏布局，表单、结果和证据互相挤压；在部分视口里关键按钮会被列内滚动藏起来。
- 候选规则只展示，不在主查询里形成“确认 `candidate_id` 后再查”的闭环。
- 运行选项部分硬编码在前端，只有 `rank_windows` 和 `sort_modes` 已接入 `/api/workbench/options`。
- 上传数据集页面能力完整，但按钮、JSON 响应和审查动作混在一起，首屏难以判断下一步。

本次改版目标是一次中等范围的主工作台重排，而不是重写推荐逻辑或后端规则管线。

## 目标

- 默认不展示 mock 数据；演示数据必须由用户显式触发。
- 让主查询按钮、数据源和运行状态在首屏常驻可见。
- 优先展示结果，减少证据面板对结果区的挤占。
- 支持主查询里的候选项确认再查，且只提交后端返回的 `candidate_id`。
- 让运行选项优先来自 `/api/workbench/options`，前端 fallback 只作为后端不可用时的保守 UI。
- 轻量整理上传与审查页面，把现有能力组织成清晰步骤。
- 保持项目边界：前端只展示 API 或 mock 输出，不生成推荐逻辑，不绕过规则验证。

## 非目标

- 不修改后端执行边界、`WorkbenchResponse` 顶层契约或规则验证策略。
- 不把城市、学费、专业候选改成 schema/value index 驱动控件。
- 不把上传数据集审查重做成字段行内批准/阻断的完整操作员系统。
- 不新增前端推断规则，不从第二轮自由文本编译 SQL 或 hard rule。
- 不删除 `demo_run.json`；它仍用于显式演示和本地汇报。

## 信息架构

页面保留三个工作区，但重新组织默认关注点：

- `查询`：默认入口，面向普通填报用户。包含表单、运行条、结果、候选确认和筛选摘要。
- `上传与审查`：上传表格、生成草稿、字段审查、批准领域、建仓、试查。
- `规则与证据`：完整规则、抽取偏好、grounding、verification、trace 和调试细节。

默认进入 `查询` 工作区。初始状态构造为空工作台状态，不再把 `demo_run.json` 写入 `runData`。结果区显示空态，提示用户填写信息后查询，并提供次要按钮 `查看演示数据`。点击该按钮后才加载 mock，并明确标记 `演示数据`。

## 主查询布局

查询页采用“顶部运行条 + 左侧表单 + 右侧结果”的结果优先双栏结构。

顶部运行条常驻展示：

- 当前数据源。
- API / 演示状态。
- 解析器、回答器和模型的当前选择。
- 主操作按钮：`开始查询`、`正在查询`、`查看演示数据` 或错误后的重试入口。

左侧表单分为三组：

- 基础信息：生源地、科类、全省排位、选考科目。
- 意向方向：专业、城市、学费上限、补充偏好。
- 受控边界：排位范围、排序方式。

排位范围和排序方式仍必须来自后端白名单。高级运行选项默认折叠，但当前选择在顶部运行条可见，避免用户不知道本次用的是哪种解析器或回答器。

右侧结果区按状态展示：

- 空态：不显示任何院校样例。
- 加载态：保持布局稳定，展示正在查询。
- 错误态：展示经 `formatApiError` 处理后的错误，不显示 traceback、路径或 secret。
- 无结果：明确说明后端返回 0 条，前端不编造推荐。
- 需要确认：展示候选项确认面板。
- 正常结果：展示结果卡片列表。

结果卡片继续展示最小有用结果形状：院校名称、院校专业组代码、专业名称、城市、学费、专业组最低位次、专业最低位次和 safety margin。

## 候选确认闭环

主查询结果如果包含 `candidates_to_confirm` 或兼容字段里的候选项，结果区上方展示 `确认后再查` 面板。

交互规则：

- 每个候选项必须绑定系统生成的 `candidate_id`。
- 用户只能勾选候选项并点击 `确认后再查`。
- 前端重新请求 `/workbench/query`，保留上一轮可复用请求参数，并提交 `confirmed_candidates: [...]`。
- 前端不得把用户自由文本二次编译成 SQL、hard filter 或确认规则。
- 对没有 `candidate_id` 的候选项只展示说明，不提供确认操作。
- `no_schema_field` 或 missing-schema 偏好即使被用户试图确认，也仍由后端拒绝；前端只展示拒绝原因。

## 运行选项与 hardcode

`/api/workbench/options` 作为 API 模式下的受控选项来源。前端使用其返回的：

- `rank_windows`
- `sort_modes`
- `extractors`
- `generators`
- `models`

前端保留最小 fallback，用于后端不可用时维持页面可渲染。fallback 必须明确处于离线或后端不可用状态，不能伪装成已从后端获取的真实选项。

保留以下前端本地常量：

- 广东项目当前默认的省份、科类和再选科目选择。
- 当前城市和学费档位的保守列表。
- 显式演示入口使用的 `demo_run.json`。
- 开发模式下的 `operator-token` 本地便利。生产构建不内置默认 token，UI 要把连接或认证状态说清楚。

## 上传与审查

上传页做轻量步骤流，保留现有后端能力：

```text
上传文件 -> 生成草稿 -> 字段审查 -> 批准领域 -> 生成可查询数据 -> 试查
```

每一步显示当前状态、可执行按钮和不可执行原因。按钮不再散落在三列面板里；不可用按钮需要通过状态或说明让用户知道缺少 dataset、草稿、审查或建仓结果。

首屏优先展示：

- 工作表与表头。
- 缺失字段。
- 风险字段。
- 审查摘要。
- 前端操作审计记录。
- 试查状态和试查结果。

原始 JSON 响应放入 `调试数据` 折叠区，默认不展开。试查结果复用主查询的状态表达：待确认、无结果、已阻断、可展示结果。

## 视觉方向

采用“面向填报的专业感”：

- 工作台密度保留，但留出比当前更多的行距和分组边界。
- 颜色克制，主要用绿色表示已执行或通过，黄色表示待确认，红色表示阻断或错误。
- 避免营销式 hero、装饰性渐变和过大的标题。
- 卡片只用于结果项、工具面板和重复项，不做卡片套卡片。
- 移动端改为自然纵向滚动，避免满屏锁滚导致按钮不可见。

## 数据流

初始加载：

```text
创建空工作台状态
-> 请求 /api/workbench/options
-> 成功：写入后端选项
-> 失败：保留 fallback 并标记后端不可用
```

API 查询：

```text
用户填写表单
-> 前端提交 user_input、hard_filters、soft_preferences、extractor、generator、model、dataset_id
-> 后端返回 WorkbenchResponse
-> 前端只展示响应中的结果、规则、候选和证据
```

显式演示：

```text
用户点击 查看演示数据
-> 前端加载 demo_run.json
-> 页面标记演示状态
```

确认再查：

```text
后端返回 candidates_to_confirm
-> 用户勾选 candidate_id
-> 前端提交 confirmed_candidates
-> 后端重新验证并执行
-> 前端展示新 WorkbenchResponse
```

## 错误处理

- API 请求失败时保留当前页面结构，不回填 mock 数据。
- 后端 options 获取失败时展示连接提示，并使用保守 fallback 渲染控件。
- 查询失败时清楚展示错误，不展示 traceback、绝对路径、`.env`、token 或 secret。
- 上传审查任一步失败时写入前端操作审计记录，并保留已完成步骤的状态。

## 测试与验收

实现完成后至少验证：

- `cd frontend && npm run build`
- 桌面视口：默认首页不展示 mock 院校，主查询按钮首屏可见。
- 桌面视口：点击 `查看演示数据` 后加载 mock，并明确标记演示状态。
- 桌面视口：结果列表、筛选摘要、证据入口不互相遮挡。
- 移动视口：页面可自然滚动，按钮、表单和结果不被隐藏。
- API options：`rank_windows`、`sort_modes`、`extractors`、`generators`、`models` 优先使用后端返回。
- 确认再查：勾选候选项后，请求体包含 `confirmed_candidates`，值来自上一轮 `candidate_id`。
- 上传页：步骤顺序清楚，JSON 调试数据默认折叠。
- 浏览器控制台无相关运行错误，页面无 Vite/Vue 错误 overlay。

如果实现改动影响 README 或 API 行为描述，需要同步更新中文文档。

## 已确认决策

- 范围选择：主工作台重排。
- 默认受众：普通查询优先，同时保留上传审查二级工作区。
- 布局方向：结果优先双栏。
- 默认数据：不默认展示 mock，演示入口显式触发。
- hardcode 清理：运行选项接 `/api/workbench/options`，不扩大到城市/学费/schema 候选。
- 候选确认：主查询支持 `candidate_id` 确认再查。
- 上传审查：轻量整理，不做完整操作员系统。
- 视觉方向：面向填报用户的专业工作台。
